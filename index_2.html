§ 06Principles

How I work.

— No. 01

Targeted patches over full rewrites.

Most production code doesn't need a rewrite. It needs precise, surgical changes that preserve everything that already works. I default to small, reviewable edits that are easy to roll back.

— No. 02

Fail visibly, never silently.

A wrong answer that looks confident is worse than a refusal. When grounding fails or inputs are missing, the system should surface that clearly — not paper over it with a plausible hallucination.

— No. 03

Backend-first, UI later.

I validate behavior in the API layer — Swagger, curl, tests — before any pixel is moved. The UI is a presentation of a system that already works, not a place to discover that it doesn't.

— No. 04

Subject-agnostic by default.

Hardcoded keyword lists and domain-specific shortcuts age badly. I prefer general approaches — embeddings, retrieval, structured prompts — that hold up across subjects, languages, and edge cases.
